Council Verdict

Public booking page UI · June 25, 2026 · detailing-app `/book/[slug]`

Question: How should we prioritize the next round of UI improvements to the public booking page — what’s high-impact vs vanity, and what should we avoid?

Chairman’s Recommendation

Run a friction-removal + trust pass, not a visual parity pass with the operator app. Keep the light consumer theme; stop porting more QuickAddJob patterns.

Where the council agrees

  • Operator-app cosplay is low ROI — customers need clarity, not CRM familiarity
  • The step 2 clock box is confusing dead UI — fix or remove first
  • Step 3 shows too many fields — collapse vehicle/location/notes with defaults
  • Trust beats aesthetics: business name, plain language, price visibility, strong confirmation
  • Reliability before polish: hydration, iOS date input, real-device testing

Where the council clashes

  • Merchandising (badges, package photos) — growth lever vs empty UI when detailers don’t configure content
  • Scope — full Calendly-style restructure vs incremental shippable slices

Resolution: Ship structural wins incrementally; only add merchandising hooks where settings data already exists.

Ship order (Executor-aligned)

  1. Fix/remove fake clock box on step 2
  2. Sticky package + price summary (steps 2–3)
  3. Copy pass: plain-language labels
  4. Collapse step 3 optional fields
  5. Upgrade confirmation screen
  6. Hide bottom nav on book routes · fix hydration
Do this first: Remove the non-interactive clock box on step 2. Show selected time only after a slot is picked — eliminates the top outsider confusion point (~30 min).

Contrarian Stop polishing — you're cosplaying the operator app

Porting QuickAddJob patterns adds cognitive load. Date picker broke twice fighting the platform. High-impact: sticky summary, tel link, empty slot states, hydration fix. Vanity: CheckCircle parity. Fatal flaw: designer satisfaction over completion rate.

First Principles Remove decisions, don't add UI

Five beats: trust → service → time → contact → confirmed. Collapse step 3 defaults. Human dates. Strong confirmation + calendar. Should feel like Calendly, not a CRM form.

Expansionist Booking link = landing page

Package merchandising, URL presets, post-booking loop. Persistent footer bar like checkout. Defer app parity; invest in detailer branding slot in settings when ready.

Outsider "This feels like a template"

Fake clock box confused me. "Fixed" means nothing. Step 3 feels like a job application. Want bigger business name, price always visible, plain language, obvious "you're done."

Executor Monday morning ship list

Half day: hydration + iOS date test + fix clock UI. Then sticky summary, copy pass, field collapse, confirmation upgrade. Do NOT build custom calendar or shared component library yet.

Peer review highlights

Strongest responses: Outsider (specific confusion), Executor (ship order).

Blind spots caught: Bottom nav on book routes, skeleton states, accessibility, slot race on submit.

Expansionist blind spot: Merchandising assumes detailers maintain photos/badges.
